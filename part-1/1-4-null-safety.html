1.4 Null Safety

Kotlin's type system eliminates NullPointerException at compile time. Learn how to handle nullable types safely.

The Problem with Null

In many programming languages, null references cause the dreaded NullPointerException (NPE). Studies suggest NPEs account for a significant portion of production bugs.

Kotlin solves this by distinguishing between nullable and non-nullable types at the compiler level.

Nullable vs Non-Nullable

In Kotlin, a regular type cannot hold null. To allow null, you must use the nullable type with ?.

Kotlin
fun main() {
    val name: String = "Kotlin"  // Cannot be null
    val nullable: String? = null  // Can be null
    
    println(name.length)      // OK
    println(nullable.length)  // Compile error!
}

Safe Handling of Nullable Types

Safe Call Operator (?.) - Returns null if the object is null, otherwise calls the method.

Kotlin
val nullable: String? = "Hello"
println(nullable?.length)  // 5

val nullValue: String? = null
println(nullValue?.length) // null

Elvis Operator (?:) - Provides a default value if null.

Kotlin
val nullable: String? = null
println(nullable?.length ?: 0)  // 0

val name = nullable ?: "Unknown"
println(name)  // Unknown

Not-Null Assertion (!!) - Use when you're certain a value isn't null (throws NPE if it is).

Kotlin
val nullable: String? = "Hello"
println(nullable!!.length)  // 5
